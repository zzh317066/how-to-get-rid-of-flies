<a name="heading_1"></a>1.1 **项目概述**


<a name="heading_2"></a>**1.1 项目名称**


暂定：

- **FlyAway Guide**
- **FlyFixer**
- **GetRidOfFlies**
- **Fly Identifier**


这个名字现在不用定死，先用一个工作名也行。

如果只是内部 PRD，可以先叫：

**Project Fly Removal**



<a name="heading_3"></a>**1.2 项目一句话描述**


一个围绕“家里 flies 清除问题”的垂直小站，结合 SEO 内容页与 AI 诊断/方案生成能力，帮助用户判断家中 flies 类型，并提供对应的清除方法与个性化方案。


<a name="heading_4"></a>**1.3 项目目标**


这个项目不是单纯的信息站，而是一个：


|**SEO 获取流量 + AI 辅助诊断 + 付费个性化方案** 的微型垂直产品。|
| :- |

目标分三层：

<a name="heading_5"></a>**第一层：获取搜索流量**

围绕高搜索量问题词获取自然搜索流量，例如：

- how to get rid of flies
- how to get rid of fruit flies
- how to get rid of drain flies
- how to get rid of house flies
- fruit fly trap


<a name="heading_6"></a>**第二层：提升用户承接**

通过站内的 AI Fly Identifier 帮助用户快速判断自己遇到的是哪种 flies，并把用户导向更精确的内容页。

<a name="heading_7"></a>**第三层：实现变现**

通过 Personalized Removal Plan 为用户生成基于家庭情况的个性化清除方案，并收取小额费用；后续也可叠加广告变现。


<a name="heading_8"></a>**1.4 项目类型判断**


这个项目不是纯博客，也不是纯工具站，更准确地说，它是：


|**内容驱动的微型垂直产品站**|
| :- |

它包含两部分：

- **内容层**：SEO 页面承接搜索需求
- **工具层**：AI 功能提升体验与转化



<a name="heading_9"></a>1.2 **为什么做这个项目**


<a name="heading_10"></a>**2.1 选题原因**


当前通过关键词调研发现，“flies removal”相关词具备以下特点：

1. **搜索量高**
2. **关键词难度相对可做**
3. **用户问题明确**
4. **搜索意图很强**
5. **内容与工具可以结合**
6. **有自然变现空间**


这意味着它不是一个“只有流量、没有承接”的词，而是适合做成一个小闭环产品。


<a name="heading_11"></a>**2.2 为什么不是单纯做一篇文章**


因为这类词背后用户的真实需求不只是“看一篇文章”，而是：

- 我家里这种 flies 到底是哪一种？
- 我应该立刻怎么处理？
- 我这种情况适合自然方法还是化学方法？
- 我家里有小孩/宠物怎么办？
- 为什么处理了还会反复出现？


所以如果只做文章，承接能力有限。

加入 AI 诊断和方案生成后，可以把通用流量变成更深层次的使用和付费。


<a name="heading_12"></a>1.3 **目标用户**


<a name="heading_13"></a>**3.1 核心用户**


英语市场中，家里出现 flies 问题、并主动搜索解决方案的家庭用户。


<a name="heading_14"></a>**3.2 用户典型场景**


<a name="heading_15"></a>**场景 A：泛问题用户**

用户家里出现飞虫，但并不确定具体类型，于是搜索：

- how to get rid of flies


此时用户需求是：

- 先搞清楚是什么
- 先知道最通用、最快的处理方式


<a name="heading_16"></a>**场景 B：明确类型用户**

用户已经大概知道是 fruit flies / drain flies / house flies，于是搜索：

- how to get rid of fruit flies
- how to get rid of drain flies


此时用户需求是：

- 获取更具体、更有效的处理方法
- 知道为什么会出现
- 知道怎么防止复发


<a name="heading_17"></a>**场景 C：想省时间用户**

用户不想自己读一堆内容整合方案，更希望：

- 直接告诉我该怎么做
- 最好根据我家情况来给建议


此时用户更适合进入 AI 工具与付费方案流程。


<a name="heading_18"></a>**3.3 用户痛点**

1. 不确定 flies 类型
2. 网上信息过于零散
3. 不知道哪种方法最有效
4. 已尝试过一些方法但无效
5. 不知道如何防止再次出现
6. 家庭条件有限制（宠物/儿童/不想用化学药剂）



<a name="heading_19"></a>1.4 **核心价值主张**


<a name="heading_20"></a>**对用户的价值**

<a name="heading_47"></a>6. **AI 功能设计**


<a name="heading_48"></a>**6.1 功能一：Fly Identifier**


<a name="heading_49"></a>**功能定位**


一个免费的 AI 识别工具，帮助用户根据家里的情况，判断自己最可能遇到的是哪一种 flies。

<a name="heading_50"></a>**目标**


解决用户最常见的第一个卡点：


|“我家里飞来飞去的到底是什么？”|
| :- |

<a name="heading_51"></a>**功能价值**


对用户：

- 降低判断难度
- 节省看文章时间
- 更快进入正确页面


对产品：

- 提高首页转化
- 增加用户停留
- 为付费方案做前置分流



<a name="heading_52"></a>**输入信息**


建议第一版输入字段尽量简单，避免太重。

用户输入：

1. Where do you see them most often?

- kitchen
- bathroom
- near sink/drain
- near trash
- around fruit
- near windows
- other

4. What do they look like?

- very small
- dark / fuzzy
- common house-fly look
- not sure

7. Where do they gather?

- fruit / food
- drain / sink
- trash
- windows
- random places

10. How severe is it?

- a few
- moderate
- many

13. Optional free-text description



<a name="heading_53"></a>**输出结果**


输出应尽量简洁，不要搞成很长的报告。

输出包括：

1. Most likely type

- fruit flies / drain flies / house flies

4. Why this is likely

- based on location + appearance + behavior

7. Recommended next step

- 链接到对应内容页

10. CTA

- Want a plan tailored to your home? Get a personalized removal plan.




<a name="heading_54"></a>**输出语气要求**


不能说成专业鉴定结论。

要用这种表达：

- “Based on your description, these are most likely…”
- “A possible match is…”
- “This tool helps narrow down the type of flies in your home.”



<a name="heading_55"></a>**6.2 功能二：Personalized Removal Plan**


<a name="heading_56"></a>**功能定位**


一个付费的 AI 方案生成器，根据用户家庭环境与 flies 类型，生成专属清除方案。

<a name="heading_57"></a>**目标**


解决用户第二个卡点：


|“我知道大概是什么了，但我不想自己读一堆文章拼方案。”|
| :- |



<a name="heading_58"></a>**功能价值**


对用户：

- 更快拿到适合自己情况的方案
- 方案更贴近家庭条件
- 减少试错成本


对产品：

- 形成直接变现闭环
- 提升站点的产品属性
- 区分“免费通用建议”和“付费专属建议”



<a name="heading_59"></a>**收费逻辑**


建议：

- Fly Identifier 免费
- Removal Plan 低价付费


价格先不定死，PRD 里先写：

- Low-ticket paid feature
- Affordable one-time payment



<a name="heading_60"></a>**输入信息**


比 Fly Identifier 更详细，但第一版仍要控制复杂度。

用户输入：

1. Confirmed / likely fly type
2. Main location
3. Severity
4. Have you tried anything already?
5. Pets or kids at home?
6. Are chemical sprays acceptable?
7. Prefer fastest result or safer/natural option?
8. Optional notes



<a name="heading_61"></a>**输出结果**


输出一个清晰、可执行的方案，而不是泛泛建议。

建议输出结构：

1. Situation Summary

- 用户当前情况摘要

4. 24-Hour Action Plan

- 今天马上做什么

7. 3-Day Removal Plan

- 接下来 3 天怎么处理

10. 7-Day Prevention Plan

- 如何防止复发

13. Recommended Supplies

- trap / cleaner / container / sealing / protective items

16. What Not to Do

- 常见误区




<a name="heading_62"></a>**输出格式要求**


要尽量像“行动清单”，而不是长篇文章。

用户付费买的是：

- clarity
- relevance
- actionability


不是买更多废话。

<a name="heading_72"></a>9. **商业模式与付费设计**


<a name="heading_73"></a>**9.1 商业模式总览**


本项目的商业模式分为两层：

<a name="heading_74"></a>**第一层：直接变现**


通过 **Personalized Removal Plan** 向用户提供个性化清除方案，收取一次性小额费用。

<a name="heading_75"></a>**第二层：流量变现**


当站点有一定搜索流量后，通过广告位实现流量型收入。


<a name="heading_76"></a>**9.2 为什么要这样设计**


这个项目的用户需求有明显分层：

<a name="heading_77"></a>**免费层需求**


用户只是想快速知道：

- 这是什么 flies
- 通用解决方法是什么
- 对应文章在哪


这部分适合通过：

- SEO 内容页
- Fly Identifier（免费）


来满足。


<a name="heading_78"></a>**付费层需求**


用户已经不满足于“看一篇通用文章”，而是更想要：

- 根据我家情况给我方案
- 我家有宠物/孩子怎么办
- 我想要自然方案，不想用化学喷剂怎么办
- 我已经试过几种办法没用，下一步怎么办
- 能不能直接给我一个按步骤执行的 plan


这部分就天然适合付费。


<a name="heading_79"></a>**9.3 第一阶段的收入优先级**


我建议第一阶段优先级这样排：

<a name="heading_80"></a>**优先级 1：个性化方案收费**


原因：

1. 更贴合用户真实需求
2. 不依赖超大流量
3. 更适合你当前“做一个小产品”的思路
4. 容易形成“免费 → 付费”的闭环



<a name="heading_81"></a>**优先级 2：广告**


原因：

1. 广告模式简单直接
2. 如果词真的有量，后续可以自然接入
3. 但广告更依赖流量规模，所以不适合作为第一阶段核心收入来源



<a name="heading_82"></a>**暂不优先：联盟推荐**


虽然理论上可以接：

- traps
- sprays
- cleaners
- pest control related products


但第一阶段不建议把重心放在联盟上，因为：

1. 会分散注意力
2. 站点前期更重要的是内容和 AI 承接
3. 联盟更适合后续在 fruit fly trap 页面自然植入



<a name="heading_83"></a>10. **免费与付费边界**


<a name="heading_84"></a>**10.1 免费提供什么**


第一阶段建议免费开放：

<a name="heading_85"></a>**A. 所有 SEO 内容页**


包括：

- 首页
- fruit flies
- drain flies
- house flies
- fruit fly trap

1. 提升首页转化
2. 帮用户分流

4. 为付费方案做前置筛选


如果一上来就把识别功能收费，会让用户阻力太大。


<a name="heading_87"></a>**10.2 付费提供什么**


第一阶段建议只对一个功能收费：

<a name="heading_88"></a>**Personalized Removal Plan**


这应当是整个项目的核心付费功能。

用户付费买到的不是：

- 更多文章
- 更长的解释


而是：

- 更适合自己家庭情况的方案
- 更节省时间的整理结果
- 更明确的执行步骤
- 更少的试错成本



<a name="heading_89"></a>**10.3 免费和付费的区分原则**


要非常清楚：

<a name="heading_90"></a>**免费 = 通用答案**


适合大多数人、覆盖一般情况的建议。

<a name="heading_91"></a>**付费 = 个性化行动方案**


根据用户输入条件调整过的、可直接执行的方案。


<a name="heading_92"></a>**10.4 免费不能弱到没价值**


一个很重要的原则是：


|免费内容必须本身就有价值，不能故意写得很烂来逼用户付费。|
| :- |

因为这个项目的流量入口主要来自搜索。

如果免费内容没有价值：

- 用户不会信任你
- 不会继续用 AI 功能
- 更不会付费


所以策略应该是：

- 免费内容：足够好，能解决大部分基础问题
- 付费方案：更省时间、更个性化、更可执行



<a name="heading_93"></a>11. **付费功能设计**


<a name="heading_94"></a>**11.1 付费功能名称（工作名）**

<a name="heading_95"></a>**11.2 付费形式**


建议第一阶段采用：

<a name="heading_96"></a>**一次性付费（one-time payment）**


不做订阅。

原因：

1. 这个需求本身更像一次性问题解决
2. 用户更容易接受
3. 产品逻辑简单
4. 更适合 MVP



<a name="heading_97"></a>**11.3 价格策略**


PRD 里先不定具体价格，但建议写清原则。

<a name="heading_98"></a>**价格原则**

- 冲动可支付
- 不需要用户深思熟虑


也就是说，它应该是一个：


|**small, low-ticket, one-time purchase**|
| :- |

而不是高价咨询产品。


<a name="heading_99"></a>**11.4 用户为什么会愿意付费**


用户会付费的原因应当建立在以下价值上：

1. **省时间**
2. 不用自己翻来翻去看不同文章再整理。
3. **更适合自己情况**
4. 有宠物 / 有小孩 / 厨房 / 浴室 / 轻度 / 重度，方案不同。
5. **更容易执行**
6. 方案不是泛泛建议，而是按时间和步骤拆解。
7. **更少试错**
8. 用户很多时候已经试过一些办法但没成功。



<a name="heading_100"></a>**11.5 付费输出物的价值感设计**

<a name="heading_104"></a>12. **广告模式设计**


<a name="heading_105"></a>**12.1 广告是否第一阶段就接**

<a name="heading_106"></a>**可以预留位置，但不要一上来重点做**


原因：

1. 前期流量不一定够
2. 一上来广告太重，会影响体验和信任
3. 你的核心差异化目前不是“内容站”，而是“内容 + AI 工具 + 付费方案”



<a name="heading_107"></a>**12.2 广告适合出现的位置**


后续有流量后，广告适合出现在：

- 内容页中段
- FAQ 前后
- trap 页面中部或底部


不建议：

- 首页 hero 区直接插广告
- AI 表单旁边塞广告
- 付费方案页插广告


因为这些地方会明显影响转化。


<a name="heading_108"></a>**12.3 广告与付费功能的关系**


广告更像“补充收入”，不是核心闭环。

项目的收入结构应优先理解为：


|SEO 流量 → 内容信任 → 免费识别 → 付费方案|
| :- |

广告只是顺带把没付费但有流量的用户做二次变现。

<a name="heading_359"></a>**28.6 Fly Identifier 技术实现**


<a name="heading_360"></a>**28.6.1 实现目标**


用户提交一个轻量表单后，系统输出：

- likely type
- why
- recommended next step
- page link



<a name="heading_361"></a>**28.6.2 第一版技术建议**


第一版不要过度复杂化。

建议逻辑分两层：

<a name="heading_362"></a>**第一层：规则判断（可控）**


先根据用户输入做一个基础分类判断：

- kitchen + fruit/food + very small → 更偏 fruit flies
- drain/sink + dark/fuzzy + bathroom → 更偏 drain flies
- windows/trash + larger common fly → 更偏 house flies


<a name="heading_363"></a>**第二层：LLM 组织语言（自然）**


把“规则判断结果 + 用户输入摘要”交给 LLM，生成：

- 自然语言解释
- why this is likely
- what to do next



<a name="heading_364"></a>**为什么建议这样做**


因为如果完全靠 LLM 自由判断：

- 输出可能不稳定
- 可能乱猜
- 可控性较差


而“规则先判断，LLM 再包装”更适合 MVP：

- 稳
- 可解释
- 容易调试

<a name="heading_414"></a>29. **内容与 SEO PRD**



<a name="heading_415"></a>**29.1 内容与 SEO 的核心目标**


本项目的内容与 SEO 目标不是“写很多文章”，而是：

1. **精准承接高意图搜索词**
2. **让每一页只负责一类清晰问题**
3. **让用户在进入页面后快速得到答案**
4. **通过结构化内容提升停留、跳转和工具使用率**
5. **避免页面之间内容重叠与关键词打架**



<a name="heading_416"></a>**29.2 内容策略总原则**


<a name="heading_417"></a>**原则 1：一页只解决一个主问题**


例如：

- 首页：how to get rid of flies
- 果蝇页：how to get rid of fruit flies
- 下水道飞虫页：how to get rid of drain flies
- 家蝇页：how to get rid of house flies
- trap 页：fruit fly trap


每页有且只有一个**主目标词**，其他词只能作为辅助覆盖。



<a name="heading_418"></a>**原则 2：先满足搜索意图，再谈转化**


每个 SEO 页面首先必须回答用户最想知道的问题。

不要一上来就：

- 强推 AI
- 强推付费
- 强塞很多 CTA


正确顺序是：

1. 先给答案
2. 再讲方法
3. 再讲区别/补充
4. 最后引导到工具或付费



<a name="heading_419"></a>**原则 3：内容必须“够用、实用、可执行”**


用户搜这类词，不是来读论文的。

他要的是：

- 我现在该怎么办
- 最快的方法是什么
- 为什么一直还有
- 哪些方法真的有效
- 下一步点哪里


所以内容必须偏：

- practical
- direct
- action-oriented



<a name="heading_420"></a>**原则 4：内容层次要清楚**


每页内容建议都遵循这个层级：

1. 快速答案
2. 问题判断/识别
3. 详细方法
4. 预防复发
5. FAQ
6. 相关页面入口
7. 工具 / 付费方案入口

<a name="heading_469"></a>**29.6 FAQ 设计规则**


FAQ 非常重要，因为它可以同时做三件事：

1. 覆盖长尾词
2. 回答用户犹豫点
3. 提高页面完整度



<a name="heading_470"></a>**FAQ 设计原则**


<a name="heading_471"></a>**原则 1：FAQ 要来自真实下一步问题**


不是随便凑 10 个问题。

而是从用户行为出发：

- 最快方法是什么
- 为什么还会回来
- 某个 DIY 方法有没有用
- 多久能解决
- 怎么判断和另一种 flies 的区别



<a name="heading_472"></a>**原则 2：FAQ 要区分页面**


不同页 FAQ 不要大量重复。

<a name="heading_473"></a>**首页 FAQ 更适合**

- Why are there flies in my house?
- What is the fastest way to get rid of flies?
- Are all flies treated the same way?


<a name="heading_474"></a>**Fruit Flies FAQ 更适合**

- Does apple cider vinegar work?
- Why do fruit flies keep coming back?
- How do I get rid of fruit flies in the kitchen?


<a name="heading_475"></a>**Drain Flies FAQ 更适合**

- Are these drain flies or fruit flies?
- Why are they in the bathroom?
- Why do they come back after spraying?



<a name="heading_476"></a>**原则 3：FAQ 回答要短、明确**


FAQ 不是正文补写，回答尽量：

- 2–5 句
- 给明确方向
- 必要时引导到对应模块

<a name="heading_491"></a>**29.9 AI 生成内容规范**

<a name="heading_492"></a>**29.9.1 AI 写作目标**

<a name="heading_493"></a>**29.9.2 AI 写作必须遵守的规则**

1. 先回答问题，再展开

4. 每个方法要解释“为什么有效”
5. 每页都必须区分于其他页的核心意图
6. 不能把 trap 页内容整块复制到 fruit flies 页
7. 不能把首页写成大而全百科页



<a name="heading_494"></a>**29.9.3 AI 写作风格要求**

- 清晰
- 直接
- 实用
- 非学术
- 非营销腔过重

- practical guide
- answer page
- step-by-step help


<a name="heading_496"></a>**不要像**

- 论文
- 堆满 SEO 同义词的垃圾文章
- 夸张营销页



<a name="heading_497"></a>**29.9.4 AI 输出结构建议**


每一页让 AI 按固定骨架产出：

1. intro / quick answer
2. what it is
3. why it happens
4. best ways to fix it
5. prevention
6. FAQ
7. internal link suggestions
8. CTA suggestion

<a name="heading_509"></a>30. **MVP 开发拆解 PRD**



<a name="heading_510"></a>**30.1 MVP 的核心目标**


这一版 MVP 不是为了把这个项目做成一个很大的平台，而是为了验证这 4 件事：

3. **用户会不会点击 Personalized Plan**
4. **整条闭环能不能跑通：内容 → AI → 支付 → 结果**


所以 MVP 的定义不是“功能很多”，而是：


|**最少功能，验证最关键的商业与产品假设。**|
| :- |



<a name="heading_511"></a>**30.2 MVP 必须验证的假设**


<a name="heading_512"></a>**假设 1：内容页能承接用户**


用户搜：

- how to get rid of flies
- how to get rid of fruit flies
- how to get rid of drain flies
- how to get rid of house flies
- fruit fly trap


进入页面后，不会立刻退出，而会继续阅读或点击下一步。


<a name="heading_513"></a>**假设 2：Fly Identifier 有真实吸引力**


如果用户不确定 flies 类型，他会愿意花 20–30 秒填写一个轻量识别表单。


<a name="heading_514"></a>**假设 3：个性化方案有付费意愿**


如果用户觉得：

- 自己情况比较特殊
- 不想自己整理方案
- 已经试过一些方法还没解决


那他会愿意点击并购买一个低价方案。


<a name="heading_515"></a>**假设 4：AI 输出结果要“像产品”，不能像废话**


如果结果只是普通一段 AI 文字，用户会觉得不值。

所以生成结果必须：

- 结构清晰
- 可执行
- 像行动计划

<a name="heading_579"></a>37. **推荐开发顺序**


这个顺序很重要，可以避免你一开始陷入太重的功能开发。

<a name="heading_580"></a>**阶段 1：站点骨架 + 内容页**


先完成：

- layout
- 首页
- 4 个子页面
- 基础 SEO
- 基础 CTA


<a name="heading_581"></a>**为什么先做这个**


因为哪怕 AI 还没完全接好，内容页本身也能成立，是流量入口。


<a name="heading_582"></a>**阶段 2：Fly Identifier**


再做：

- 表单
- 基础规则判断
- 结果展示
- 存库


<a name="heading_583"></a>**为什么第二个做这个**


因为它是最轻量、最能提升“产品感”的功能。


<a name="heading_584"></a>**阶段 3：Personalized Plan 表单 + 支付**


再做：

- plan 表单
- checkout
- success/cancel
- request 记录


<a name="heading_585"></a>**为什么先不急着做完整生成**


你可以先把支付与流程搭通，再做生成逻辑。


<a name="heading_586"></a>**阶段 4：Plan 生成 + 结果页**


最后接：

- AI 结果生成
- 结果保存
- 结果展示页



<a name="heading_587"></a>**阶段 5：埋点 + 小修**


把：

- 埋点
- CTA 文案
- 页面小优化
- 补上。


<a name="heading_588"></a>38. **每个阶段的“可停点”**


这个很重要，因为你可能中途状态起伏，或者时间不够。

<a name="heading_589"></a>**停点 1：内容页完成**


即使 AI 还没做，至少站已经能上线承接流量。

<a name="heading_590"></a>**停点 2：Fly Identifier 完成**


即使付费还没做，至少站已经有点工具感和分流能力。

<a name="heading_591"></a>**停点 3：付费闭环完成**


到这里就是一个完整 MVP。


|也就是说，你不是“必须一口气做完所有东西”，而是可以分层交付。|

<a name="heading_598"></a>41. **当前阶段最重要的产品取舍原则**


如果你后面开发时开始“想加很多东西”，用这条来砍需求：


|**凡是不能直接帮助我们验证“流量能不能进来、AI 能不能承接、用户愿不愿意付费”的功能，第一版都先不做。**|
| :- |

所以第一版不要被这些分散：

- 炫酷动画
- 复杂账号系统
- 太多物种扩展
- 图片识别
- 各种 fancy dashboard
